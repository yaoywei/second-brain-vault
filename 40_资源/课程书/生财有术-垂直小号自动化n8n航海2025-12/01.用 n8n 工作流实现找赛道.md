## 垂直小号自动化（n8n 工作流）丨实战手册丨2025年12月航海

## 01.用 n8n 工作流实现找赛道

💡

在上一章完成了 n8n 的环境部署后，我们已经拥有了强大的自动化引擎。现在，是时候让引擎转动起来，真正切入业务场景了。对于做垂直小号来说，最痛苦的莫过于：不知道选什么赛道、不知道今天写什么、写文章太耗时。

本章将带你构建一套完整的“自动化内容中台”，利用 n8n 将飞书多维表格、微信公众号 API、极致了数据以及 DeepSeek 大模型串联起来。我们将不再手动搜索和复制粘贴，而是让数据自动流转，让 AI 替你完成从调研到创作的 80% 重复性工作。

在本章中，你将亲手实践：

•

数据化选品定赛道：不再拍脑袋决定写什么，而是通过配置 API 和循环节点，批量抓取并分析公众号数据。利用 AI 自动打标和清洗数据，筛选出真正有潜力的对标账号，建立你的“赛道情报库”。

•

搭建自动更新选题库：告别选题焦虑，搭建两套监控系统——“日报”监控对标账号动态，“周报”抓取搜一搜热点。系统将自动把热门选题推送到你的飞书表格，让你每天都有写不完的灵感。

•

实现全自动二创发布：打通内容生产的“最后一公里”。从读取飞书素材开始，让 AI 自动完成标题与正文的深度二创，调用 DALL-E 3 生成封面图，并自动上传至微信公众号草稿箱。你只需做最后的审核，即可一键发布。

通过本章的学习，你将不再是一个单纯的内容创作者，而是一个拥有一支“AI 数字化团队”的管理者。让我们一起动手，把繁琐的运营工作交给 n8n，把时间留给真正的思考与决策。

社区节点是由 n8n 开发者社区贡献的扩展功能包（也就是官方没有的节点）。在本案例中，我们需要安装 Feishu （Lite） 和 Variable Storage 这两个非官方内置节点。

Feishu （Lite） 用于打通飞书消息通知的通道，Variable Storage 用于在工作流中进行更灵活的数据存储与变量管理。

注意：社区节点仅支持本地部署（Self-hosted）或服务器托管版本的 n8n，官方云端版（Cloud）暂不支持此操作。

我们需要构建一个标准化的数据“容器”，用来接收和存储 n8n 抓取到的数据。同时，必须正确获取这个容器的“地址”（ID）并授权给我们的应用，确保数据能准确无误地写入，就是多维表格

具体配置操作（如下图所示）：

a.创建多维表格

点击飞书左侧导航栏的【云文档】，选择【云盘】，进入【我的文件夹】，点击页面右上角的【新建】并选择【多维表格】

这一步至关重要，必须在“云文档”而非“知识库”中创建，因为只有云文档生成的 URL 包含 base 字段，才符合 n8n 飞书节点的调用标准，而知识库（Wiki）生成的 URL 结构不同，会导致 API 调用失败

d.记录多维表格 base 和 table 对应的字符串 token，在 n8n 工作流中需要用到

查看浏览器上方多维表格的地址栏 URL，提取关键的身份代码。

我们需要找到 /base/ 和? 之间的字符串作为 Base ID（文档 ID），以及 table= 和 & 之间的字符串作为 Table ID（数据表 ID）（具体如下图所示）

这两个 ID 就像是具体的“门牌号”和“房间号”，在后续 n8n 配置中必不可少，系统只有靠它们才能精确定位到要写入数据的位置

下面的不是代码，是 base 和 table 对应的 token 串，在 n8n 工作流中，添加飞书节点，添加记录、更新记录和查询节点时都需要用到

```
# 创建的多维表格的URL中会包含类似下面的字符串
base/JgvdboSNvaOfJHs35gQcaENRnAg?table=tblcOLnylLi1S5r4&view=vewLdXA14Q
# 其中base是(/到?中间的部分)
JgvdboSNvaOfJHs35gQcaENRnAg
# table是(=到&中间的部分)
tblcOLnylLi1S5r4
```

【Set variables】 → “写记事本”

【Get variables】 → “读记事本”

接下来我们要构建的循环链路，步骤是这样的：读记事本 → 发送HTTP请求 → 写记事本

这就产生了一个问题：循环第一次启动时，这个“记事本”是空的。如果你直接去“读”，什么都读不到，后面的HTTP请求就会因为缺少参数而失败。所以，我们现在要做的事，就是“初始化”：在循环开始之前，先往“记事本”里写上第一笔。这样，当循环启动时，第一步“读记事本”就能读到内容，整个流程就能顺利地跑起来了。

2.3.5 节点五：Loop Over Items （循环） 节点

这个节点负责批量处理与结果聚合。它的工作机制是接收上游传入的数据列表（例如 10 个页码），针对列表中的每一项数据，逐次执行后续的流程链路。当所有循环执行完毕后，它会将每一次执行的结果汇总起来，通过 Done 端口一次性输出。

```
// 从输入数据中解析 cookies 并提取 cookies_buffer 字段
const inputData = $input.all();

const result = inputData.map(item => {
  try {
    // 解析 cookies 字符串为 JSON 对象
    const cookiesObj = JSON.parse(item.json.cookies);
    
    // 返回包含 cookies_buffer 字段的对象
    return {
      json: {
        cookies_buffer: cookiesObj.cookies_buffer
      }
    };
  } catch (error) {
    // 如果解析失败，返回错误信息
    return {
      json: {
        error: 'Failed to parse cookies',
        message: error.message
      }
    };
  }
});

return result;
```

2.3.12 节点十二：Code （数据拆分） 节点

这个节点主要用于重组数据结构。Loop 节点输出的数据通常是按批次分组的（例如 2 次循环，每次 15 条，结果是包含 2 个大数组的数据包），这种嵌套结构不方便后续节点逐条处理。通过这段代码，我们将这些嵌套的数据“拆开”并合并，变成平铺的 30 条独立记录，以便后续节点能针对每一篇文章单独进行操作。

![](https://sphere-sh.oss-cn-shanghai.aliyuncs.com/private/docx/drive/media/Y7jCbrqjcoq4rHxtFo3cjfCNnrh_hhkc?Expires=1784270617&OSSAccessKeyId=LTAI5tDANFnEjK2qWnHwZKzr&Signature=eWIv6ZidUlqn5I8fDWLvVgHBpuQ%3D)

点击 Loop Over Items 节点右侧的 Done 端口，搜索并添加 Code in JavaScript 节点

在代码输入框中填入下方代码，这段代码的逻辑是遍历所有批次的数据，将其中的文章信息全部提取出来，合并成一个包含 30 篇文章的单一列表

```
// 从输入数据中提取 data 数组
const inputData = $input.all();
let items = [];

// 标记是否至少发现过一个 data 数组
let hasDataArray = false;

for (const item of inputData) {
  const json = item.json || {};

  if (Array.isArray(json.data)) {
    hasDataArray = true;

    for (const dataItem of json.data) {
      // dataItem.items 才是你截图里的那个 items
      if (!Array.isArray(dataItem.items)) {
        continue;
      }

      // 过滤掉 source 里没有 title 的 items
      const filteredInnerItems = dataItem.items.filter(inner => {
        const src = inner && inner.source;
        const title = src && src.title;
        return title && String(title).trim() !== '';
      });

      // 如果这个 dataItem 里所有 items 都被过滤掉，就跳过整个 dataItem
      if (filteredInnerItems.length === 0) {
        continue;
      }

      // 用过滤后的 items 覆盖原来的 items，其它字段保持不变
      const newDataItem = {
        ...dataItem,
        items: filteredInnerItems,
      };

      // 和你原来的逻辑一样，把每个 dataItem 展开成一条输出
      items.push({
        json: {
          ...newDataItem,
          processed_at: new Date().toISOString(),
        },
      });
    }
  }
}

// 如果根本没有 data 数组，就返回原始输入（保持原有兜底行为）
if (!hasDataArray) {
  return inputData;
}

return items;
```

2.3.15 节点十五：Code （HTML 清洗） 节点

这个节点主要用于文本格式转换。上一步获取的内容是包含各种代码标签（如 div， style）的网页源代码，这些代码对 AI 分析赛道没有帮助，反而会消耗大量的 Tokens（费用）。这段代码的作用是将 HTML 清洗为纯文本，只保留人类可读的文字内容。

![](https://sphere-sh.oss-cn-shanghai.aliyuncs.com/private/docx/drive/media/GJqPb9HPqowxidx0llGcdTrKnmT_hhkc?Expires=1784270617&OSSAccessKeyId=LTAI5tDANFnEjK2qWnHwZKzr&Signature=jDZm53JVTbbgCnKb5mkGF2TqWn8%3D)

节点配置操作（如下图所示）：

点击 HTTP Request 节点右侧的 + 号，选择 Code in JavaScript

复制并粘贴下方代码，这段脚本会利用正则表达式移除所有的网页标签、脚本和样式，只留下纯净的文本。

因为上一步返回的文章是 html 网页结构的，里面带有很多网页元素的标签，不利于大语言模型理解。使用 code 节点把它转换成纯文本，把它作为 AI agent 节点的用户提示词

```
// 获取所有输入内容
let items = $input.all();

// 处理每个输入项
let results = items.map(item => {
  let content = item.json.content;
  
  if (typeof content === 'string') {
    let text = content;
    
    // 1. 移除 script 和 style 标签及其内容
    text = text.replace(/<script\b[^<]*(?:(?!<\/script>)<[^<]*)*<\/script>/gi, '');
    text = text.replace(/<style\b[^<]*(?:(?!<\/style>)<[^<]*)*<\/style>/gi, '');
    
    // 2. 移除所有HTML标签
    text = text.replace(/<[^>]*>/g, '');
    
    // 3. 解码HTML实体
    text = text
      .replace(/&nbsp;/g, ' ')
      .replace(/&amp;/g, '&')
      .replace(/&lt;/g, '<')
      .replace(/&gt;/g, '>')
      .replace(/&quot;/g, '"')
      .replace(/&#39;/g, "'")
      .replace(/&#x27;/g, "'")
      .replace(/&#x2F;/g, '/');
    
    // 4. 清理多余空白字符
    text = text
      .replace(/\s+/g, ' ')           // 多个空格合并为一个
      .replace(/\n\s*\n/g, '\n\n')    // 保留段落换行
      .trim();
    
    return { 
      json: { 
        cleanText: text, 
        originalLength: content.length, 
        cleanedLength: text.length 
      } 
    };
  } else {
    return { 
      json: { 
        cleanText: content, 
        error: 'Input was not a string' 
      } 
    };
  }
});

return results;
```

2.3.18 最终优化：错误处理与正式运行

这一步主要用于提高工作流的容错性。在处理大量数据时，难免会遇到个别 API 请求失败或文章被删的情况。设置错误处理机制可以确保“坏数据”被跳过，而不会导致整个自动化流程中断。

配置操作：

分别双击打开 两个 HTTP Request 节点和 Feishu 节点，点击 Settings 标签页，将 On Error 选项修改为 Continue，这一步意味着当该节点执行出错时，忽略错误并继续执行下一个数据项的操作

1.

找到第二个 HTTP Request 节点（获取详情），关闭 Settings 里的 Execute Once 开关，这一步是恢复全量数据处理能力

整个系统基于 n8n 实现，旨在将上一节中“粗选”出来的账号，通过接入第三方数据平台（极致了 API），自动补全粉丝数、阅读量、发文周期等核心数据，从而实现自动化的筛选与决策。核心流程如下：

启动与数据获取 流程由 When clicking ‘Execute workflow’ 节点手动触发。 启动后，Feishu 节点会一次性拉取上一节中我们存储在飞书多维表格里的所有账号记录（例如 29 条数据）。

数据清洗与预处理 飞书返回的数据通常是一个包含所有记录的“大包裹”。为了能逐个处理账号，我们需要使用 Code 节点将这个“大包裹”拆解（Split Out）成 29 个独立的“小包裹”，每个包裹仅包含我们需要用到的“公众号名称”和“Record ID”。 在正式跑全量数据前，通过 Limit 节点进行“截断”，仅允许 1-2 条数据通过，用于低成本测试 MVP（最小可行性产品）流程。

核心循环：数据丰富化（Loop 机制） 进入 Loop Over Items 循环控制器，对每一个账号进行独立的 API 查询，直到所有账号处理完毕。在循环内部，数据流转分为三步走：

第一步：基础数据获取 调用 HTTP Request 节点（极致了 API），根据“公众号名称”获取该账号的粉丝数、平均阅读量、平均点赞数、周发文量等基础指标。

第二步：深层数据挖掘（首发时间） 为了判断账号的运营时长，我们需要找到它的“第一篇文章”。 首先，调用 HTTP Request 1 获取该账号的历史发文列表，从中提取出“总页数”（Total Page）。 然后，利用提取出的总页数作为参数，发起 HTTP Request 2 请求，直接定位到历史文章的“最后一页”，抓取该页中最后一条数据的时间，即为该账号的“首次发文时间”。

第三步：数据回写与归档 将上述步骤中获取到的 7 个关键指标（粉丝数、阅读、点赞、周发文、首发时间等）通过 Edit Fields 节点进行汇总打包。 最后，使用 Feishu的“更新记录”功能，利用 Record ID 准确地将这些数据填回到飞书表格对应的行中。

安全与容错机制 为了防止高频请求导致 API 封控，我们在循环末尾添加了 Wait 节点，让工作流每处理完一个账号就“休息” 1 秒。 同时，针对极致了数据库可能未收录某些账号的情况，我们将 HTTP 请求节点设置为“On Error Continue”，确保个别账号查询失败不会导致整个工作流崩溃。

温馨提示：如果你想要快速搭建好这个工作流，可以到案例最后的【直接拖动 JSON 搭建工作流】这部分内容，通过拖动 JSON 文档+配置关键节点的方法，快速搭建；

3.6 用飞书多维表格的触发器自动选对标

到这里，所有利用 n8n 工作发起 http 请求获取对标账号的基础信息的操作就都完成了。最后一个单选字段“是否”对标的信息的自动选择，有两种方法，可以用 n8n 工作流，也可以用飞书自带的自动化触发器。接下来，看下如何利用飞书应用自带的功能来自动做选择。本来这一步是在一开始建立飞书表格的时候就可以做的。但是有了这些具体的数据，理解触发器会比较直观一些。
