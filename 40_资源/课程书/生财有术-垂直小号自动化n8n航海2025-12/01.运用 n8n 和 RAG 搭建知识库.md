## 垂直小号自动化（n8n 工作流）丨实战手册丨2025年12月航海

## 01.运用 n8n 和 RAG 搭建知识库

在本节中，先以一个具体的例子把 RAG（检索增强生成）的技术原理讲明白，弄清楚什么是嵌入模型、什么是向量数据库，掌握了这些概念以后，制作基于知识库的公众号文章生成工作流，当然也可以将 rag 技术应用在其他的一些业务场景。

在人工智能的浪潮中，大语言模型（LLM）如 GPT 系列展现了令人惊叹的对话和生成能力。然而，它们存在两个固有的瓶颈：知识的滞后性（训练数据有截止日期）和 “幻觉”问题（可能编造看似合理但错误的信息，细分领域的专业度不足）。RAG（检索增强生成）技术，正是为了解决这些问题而诞生的其中一种革命性架构。

没有 rag 和外部工具的情况下，用户是这样跟大语言模型这样交互的：

![](https://sphere-sh.oss-cn-shanghai.aliyuncs.com/private/docx/drive/media/AVGhbxnhdod1kWxFYYNc70utnQc_hhkc?Expires=1784292692&OSSAccessKeyId=LTAI5tDANFnEjK2qWnHwZKzr&Signature=3T8f0JbyXA%2FkhDQh9gr2JA7eVeI%3D)

惠普出了一款打印机：DeskJet 2700，假设它是在大语言预训练之后上市销售的，那大语言模型是没有任何关于这个产品的信息的，当用户提到跟这个产品相关的问题时，“无法打开嵌入式 web 服务，怎么办？”，大语言模型是不知道答案的，结果要么回答不知道，要么胡编乱造一个答案。

如果把打印机的产品手册加上用户的问题一起发送给大模型能不能解决问题呢？答案是可以！但是这样做有两个问题，第一个大语言模型是有上下文窗口长度限制的，如果产品手册内容非常多，多到大于上下文长度，怎么办？第二个问题：就算大语言模型的上下文窗口足够长，长到能放下任何的产品手册，这么做也是高成本和低效的，token 是要耗费算力的，而算力是要钱的，没有必要几段文字就能解决的事情，硬是要丢一本《辞海》给大模型让它找答案。

那有没有一种办法能够自动的找到那些跟用户问题有关的文字片段，然后把用户问题和文字片段一起发送给大模型，让大模型给出正确的回答。RAG 就是基于这样的思想产生的。

2.RAG 的原理和流程：从“凭记忆回答”到“先查资料再回答”

想象一下，一位学识渊博但记忆停留在去年的老教授（代表基础大模型），与一位配备了最新图书馆检索系统并能在秒级内找到任何相关文献的助理（代表 RAG 系统）之间的区别。RAG 模拟了后者，它包含三个核心步骤：

1.

检索：

◦

当用户提出问题时，RAG 首先不会让模型直接回答。

◦

它会利用一个嵌入模型，将用户的问题转换为一个数学向量（即一段数字序列，可以理解成文本的“数学指纹”）。

◦

然后，这个“问题向量”会在一个庞大的向量数据库中进行相似度搜索。这个数据库里存放着各种外部知识源（如公司文档、产品手册、最新新闻、行业报告等）被预先处理成的向量。

2.

增强：

◦

系统从数据库中检索出与当前问题最相关的几个知识片段（文本块）。

◦

将这些检索到的、最新的、权威的“上下文”信息，与用户的原始问题组合在一起，形成一个新的、信息更丰富的提示词。

3.

生成：

◦

将这个增强后的提示词发送给大语言模型。

◦

大模型基于它强大的理解能力和逻辑推理能力，结合刚刚提供的上下文来生成最终答案。

💡

简而言之，RAG 的核心原理是：不要只问模型“你知道什么”，而是告诉它“这些最新的、确凿的资料，结合你的知识，应该如何回答”。

这里特别要注意，不同的嵌入模型的维度是不一样的，Qwen3-embedding:4b 的是 2560，在 4.1 的图里可以看到，如果使用的是其他的嵌入模型，一定要填写对相应的维度数

```
-- 1. 启用向量扩展
create extension if not exists vector;

-- 2. 创建文档表
create table documents (
  id bigserial primary key,
  content text,
  metadata jsonb,
  embedding vector(2560) -- Qwen3-embedding:4b的嵌入维度
);

-- 3. 创建文档相似度搜索函数
create or replace function match_documents (
  query_embedding vector(2560), -- Qwen3-embedding:4b的嵌入维度
  match_count int default 10,
  filter jsonb DEFAULT '{}'
) returns table (
  id bigint,
  content text,
  metadata jsonb,
  similarity float
)
language plpgsql
as $$
begin
  return query
  select
    d.id,
    d.content,
    d.metadata,
    1 - (d.embedding <=> query_embedding) as similarity -- 余弦向量相似比较
  from documents d
  where d.metadata @> filter
  order by d.embedding <=> query_embedding
  limit match_count;
end;
$$;
```

如果这里不是太理解建表和匹配逻辑也不要紧，后面看具体的例子回来再来看就能明白其中大概的含义。

文本分割：这个选项非常重要，默认的 simple 是每块 1000 个字符，块与块间重叠 200 个字符，什么意思呢？做向量化其实是把原始文件切割成一小块一小块的，比如一个文件有 9500 个字符，如果一块 1000 个字符，不做重叠，就会有 10 块，前面的 9 块都是 1000 字符，最后一块 500 个字符，一共 10 块。但是一般需要做重叠，比如“确保充电器与手表接触良好”这样的一个句子，如果"电"这个字正好是第 1000 个字符，“器”是 1001 个字符，如果不做重叠，完整的一个词就被切割开了，所以一般需要做重叠一下，也就是说第一块从 1-1000 个字符，第二块从 801-1800 个字符，中间重叠了 200 个字符，第三块、第四块以此类推，为了就是保持语义的完整性。

RAG 的出现，极大地提升了 AI 应用的实用价值和可靠性：

提升答案的准确性与事实性：通过提供权威依据，从根本上减少了模型“胡编乱造”的可能，让回答更加精准可靠。

打破模型的知识时效壁垒：企业的最新动态、产品的更新日志、当天的新闻，都可以通过更新知识库来让模型获取，使其答案永不过时。

赋予专业领域知识：无需耗费巨资从头训练一个专业模型，只需将内部文档、技术手册、法律法典等构建成知识库，RAG 就能让通用大模型瞬间变身领域专家。

增强透明性与可信度：RAG 系统可以引用其答案的来源（如“根据 XX 文档第 Y 节”），让用户可以追溯和验证信息的真实性，建立信任。

降低成本与门槛：相比于为每个特定任务微调大模型，构建和更新一个 RAG 知识库的成本要低得多，让更多企业能够享受到大模型的能力。

第 9 个节点：http 请求节点（公众号新增草稿 api）

```

其中title来自agent节点的输出，content来自agent1的输出，thumb_media_id来自set节点的输出
```
