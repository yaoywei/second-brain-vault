这个节点是工作流的手动启动器。与自动定时触发不同，这个节点允许我们在搭建和调试阶段，通过点击按钮来随时测试整个流程是否畅通。在开发初期，我们通常使用这个节点来手动控制执行，方便观察每一步的运行结果。

3.1.2 节点二：Feishu （读取多维表格） 节点

这个节点主要用于从飞书多维表格中批量获取数据。我们需要将事先收集好的原始文章素材读取到工作流中，作为后续 AI 改写和处理的基础输入源。通过配置这个节点，系统可以自动连接到指定的飞书表格，并拉取所有记录，确保自动化流程有内容。

节点配置操作（如下图所示）：

点击 Trigger 节点右侧的 ➕ 号，搜索并选择 【Feishu Node】，选择【多维表格 查询记录】。这一步旨在添加一个能够与飞书生态系统交互的组件。

在 【应用级别凭证】 下拉菜单中选择已配置好的 Feishu Credentials account。这一步是验证身份，确保 n8n 拥有访问你飞书文档的权限。

将 【Resource】 设置为 多维表格，并将 【Operation】 选择为 查询记录。这一步明确告知系统我们要执行的操作是“读取”多维表格中的多行数据，而不是新增或修改。

在 【多维表格 Token】 和 【多维表格 ID】 输入框中，分别填入你目标表格的对应 ID（通常在飞书表格 URL 中可以找到）。这一步是核心定位配置，它告诉系统去哪一个具体的文档和子表中抓取数据。

将 【分页大小】 设置为 20（或者根据你的需求调整）。这一步用于控制单次拉取的数据量，避免一次性请求过多数据导致超时。

请全体复制下面的即可：

```
{"filter":{
  "conjunction": "and",
  "conditions": []
}}
```

3.1.3 节点三：Split Out（拆分数据） 节点

这个节点主要用于将批量获取的数据包拆解成独立的单条记录。上一步从飞书拉取的数据通常是一个包含多篇文章的大数组（List），就像一箱打包好的苹果。为了让后续的 AI 节点能够逐一处理每一篇文章，我们需要先用这个节点把箱子打开，把苹果一个个拿出来，让工作流从“批处理”模式切换为“单项处理”模式。

节点配置操作（如下图所示）：

点击 Feishu 节点右侧的 ➕ 号，搜索并选择 【Split Out】。这一步旨在引入一个能够对数组数据进行“解包”的工具。

【Operation】：默认配置即可，在fields to split out部分填入需要拆分的数组即可

在 【Fields To Split Out】 输入框中，填入 data.items。这一步是明确告诉系统，哪一个字段里包裹着我们需要拆分的文章列表。

```
data.items
```

配置理由： 只有将“一包数据”拆散成“一条条数据”，后续的 AI 改写、发布等节点才能针对每一篇文章独立运行，否则系统会把整个列表当作一篇文章处理，导致逻辑错误。

3.1.4 节点四：IF （逻辑判断） 节点

这个节点主要用于设置过滤条件，确保自动化流程只处理我们想要的任务。为了避免重复改写已经处理过的文章，或者处理那些信息不全的无效数据，我们需要在这里设一道“关卡”，只放行符合特定标准的数据进入后续流程，用来判断是否为'未改写'

节点配置操作（如下图所示）：

点击 Split Out 节点右侧的 ➕ 号，搜索并选择 【If】。这一步旨在添加一个能够根据条件对数据进行分类处理的组件。

在 【Conditions】 区域，我们需要设置两条必须同时满足的规则（AND 关系）。

首先点击第一个条件框，点击表达式按钮选择 {{ $json.fields.Status }}，将判断符设置为 is equal to，并在值中填入 未改写。这一步是筛选出工作状态为待处理的任务。

```
{{ $json.fields.Status }}
未改写
```

接着点击下方的 Add Condition 按钮添加第二条规则。

点击表达式按钮选择 {{ $json.fields\['参考文章链接'\].link }}，将判断符设置为 is not empty。这一步是确保每一条通过的数据都包含真实有效的 URL 链接，防止后续抓取网页时出错。

```
{{ $json.fields['参考文章链接'].link }}
```

配置理由： 只有同时满足“状态是未改写”且“有文章链接”这两个条件的记录，才会通过 True 分支继续向下执行；不满足条件的记录会被自动拦截，从而避免了资源的浪费和错误的发生。

3.1.5 节点五：Code （数据清洗与合并） 节点

这个节点主要用于对飞书表格中的原始数据进行标准化处理。飞书读取出来的文章正文往往包含多行文本、HTML 标签或者多余的空行，这样的“脏数据”直接喂给 AI 可能会导致生成效果不佳。通过这段 JavaScript 代码，我们将正文中的所有段落提取出来，去除杂质，并合并成一个整洁的纯文本字段，确保 AI 能够准确理解文章内容。

节点配置操作（如下图所示）：

点击 If 节点（True 分支）右侧的 ➕ 号，搜索并选择 【Code】。这一步旨在添加一个自定义脚本组件，用于执行复杂的数据清洗逻辑。

在 【Mode】 下拉菜单中选择 【Run Once for All Items】。这一步告知系统我们需要对筛选通过的所有文章记录进行批量的格式化处理。

在 【Language】 中选择 JavaScript，然后在下方的代码输入框中，复制并粘贴下方提供的代码。这段代码的核心逻辑是遍历每一条记录，提取正文内容，利用正则表达式去除多余的空行和特殊符号，最后将处理好的干净文本赋值给一个新的字段 正文\_合并。

```
return items.map(item => {
  const blocks = item.json.fields?.['正文'] ?? [];
  const text = blocks
    .map(b => {
      // 兼容不同富文本块结构
      if (typeof b === 'string') return b;
      if (b?.text) return b.text;
      if (b?.plain_text) return b.plain_text;
      if (Array.isArray(b?.content)) {
        return b.content.map(c => c.text ?? '').join('');
      }
      return '';
    })
    .join('')                    // 想每段落换行就改成 .join('\n')
    .replace(/\n{3,}/g, '\n\n')  // 清理多余空行
    .trim();

  item.json['正文_合并'] = text;
  return item;
});
```

3.1.6 节点五：Edit Fields （设置变量） 节点

这个节点主要用于数据的提取与重命名。上一步的代码节点已经把正文清洗好了，但标题还藏在飞书复杂的原始数据结构里。我们需要用这个节点把“清洗后的正文”和“原始标题”单独提取出来，并给它们起个简单易懂的新名字，方便后续的 AI 节点直接调用。

节点配置操作（如下图所示）：

点击 Code 节点右侧的 ➕ 号，搜索并选择 【Edit Fields】。这一步旨在添加一个能够对数据字段进行自定义编辑和赋值的工具。

在 【Mode】 下拉菜单中选择 【Manual Mapping】。这一步明确告知系统我们要手动指定保留哪些字段，而不是继承所有杂乱的数据。

在 Fields to Set 区域，我们需要配置第一个字段。在 【Name】 输入框中填入 正文内容，然后在 【Value】 输入框中点击表达式按钮，填入 {{ $json\['正文\_合并'\] }}。这一步是直接引用上一个代码节点清洗好的纯文本数据，并将其重命名为“正文内容”。

```
{{ $json['正文_合并'] }}
```

接着点击 Add Field 添加第二个字段。在 【Name】 输入框中填入 原文标题，在 【Value】 输入框中点击表达式按钮，填入 {{ $json.fields\['标题'\]\[0\].text }}。这一步旨在将原本包裹在飞书深层结构里的标题文本“挖”出来，变成一个扁平化的字段，方便后续使用。

```
{{ $json.fields['标题'][0].text }}
```

3.1.7 节点六：Loop Over Items （循环） 节点

这个节点主要用于控制数据处理的节奏和批次。虽然数据在前面已经被拆分成了独立的记录，但为了防止瞬间向 AI 服务发送过多请求导致触发频率限制（Rate Limit）或接口报错，我们需要在这里强制“降速”。通过配置这个节点，我们可以让工作流从“一拥而上”变为“逐个排队”，确保每一篇文章都能被稳定、有序地送入 AI 进行改写。

节点配置操作（如下图所示）：

点击 Edit Fields 节点右侧的 ➕ 号，搜索并选择 【Loop Over Items】。这一步旨在引入一个批次控制器，确保后续的高负载任务能够串行执行。

在 【Batch Size】 输入框中填入 1。这一步是整个稳健性的核心配置，它明确告知系统每次循环只允许通过 1 条数据。这意味着不管前面筛选出了多少篇文章，到了这里都会被强制排队，处理完一篇才会放行下一篇。

理由：强制工作流每次只处理一篇文章，若表格中有多行数据（多篇文章）未改写，处理完这一篇并上传成功后，再回头处理下一篇，串行循环进行

注意事项：后续需要虚幻的部分均应放置于循环节点的replace me 部分（通过loop分支），后续所有节点都需要放置在replace部分

3.2 模块 2 - 公众号标题、正文、摘要二创

3.2.1 节点七：AI Agent （内容选题方向分析） 节点

这个节点是整个创作流程的“大脑”，负责对原始素材进行深度的战略分析。在直接动手改写之前，我们需要先让 AI 像一位资深的内容总监一样，审视这篇文章的核心价值、受众匹配度以及潜在的爆款点。通过这一步的“思考”，我们可以为后续的写作定下精准的基调，确保产出的内容不仅流畅，而且有深度、有观点。

节点配置操作（如下图所示）：

点击 Loop Over Items 节点右侧的 ➕ 号，搜索并选择 【AI Agent】。这一步旨在引入一个具备高级推理能力的智能体，用于执行复杂的文本分析任务。

在 【Source for Prompt】 中选择 【Define below】。

在 【Prompt】 输入框中，我们需要填入动态的输入内容。点击表达式按钮，

分别填入： ## 原文标题： {{ $json\['原文标题'\] }}#；# 原文内容： {{ $json\['正文内容'\] }} 这一步是将我们在前面 Edit Fields 节点中整理好的标题和正文传递给 AI。

```
## 原文标题：
{{ $json['原文标题'] }}
## 原文内容：
{{ $json['正文内容'] }}
```

在 【System Message】 输入框中，请务必完整复制并填入下方的长文本提示词。这段提示词极其关键，它详细定义了 AI 的角色（公众号首席策略官）、分析逻辑（评分、价值提炼、机会分析）以及严格的输出格式。

```
## 角色:
公众号首席内容策略官兼信息分析师

你是一个信息处理和内容策略专家。你的任务是接收一段已经过初步筛选的、与我IP定位高度相关的原始信息，并对其进行深度分析，规划出具有爆款潜力的内容策略。

## 工作流程与分析逻辑:

### 第一步：潜力评分
* 你需要根据以下标准对选题潜力进行综合评分（满分10分）。
    * **价值深度 (1-4分):** 能否从信息中提炼出有价值、可落地的观点或教程？
    * **时效性/新颖性 (1-3分):** 是不是最新的新闻、独特的视角或稀缺的信息？
    * **爆款潜力 (1-3分):** 是否有潜力结合人性钩子进行放大传播？

* 问自己一个问题：“这段信息里，最让我的读者眼前一亮、觉得‘有用’或‘有料’的一句话是什么？”
* 将这个核心信息点提炼出来。

### 第三步：机会分析与选题建议
* 基于提炼出的核心信息，快速判断这个角度可能触及到哪些人性吸引力钩子（如：恐惧、贪婪、懒惰、猎奇等）并进行简短分析。

## 输出格式:
* 请严格按照以下范例的结构，直接输出分析结果，不要包含任何额外的解释性文字。

---
### **输出格式范例**

**选题：** [此处填写原文的标题]

**1. 基础信息**
    **原文标题：**  [直接粘贴原文标题再次] 
    **原文摘要（≤100字）：**  [总结原文核心信息，不赘述、不超过100字]
    **IP 支柱：** [n8n自动化 / AI前沿发展 / 个人成长与创业]

**2. 潜力与价值分析**
    **潜力评分：** [例如：9/10]
    **评分理由：** [基于价值深度、时效性、爆款潜力的简要说明]
    **核心信息：** [用一句话概括对目标读者最有价值的信息点]
    **人性钩子：** [恐惧, 懒惰, 贪婪]
    **钩子分析：** [恐惧（不学习就会被AI淘汰）、贪婪（利用AI搞钱的新思路）]

**3. 选题方向**
    选题切入点：[**请基于核心价值，从以下三个维度筛选一个最相关的、最具吸引力的选题创作方向或角度，你的目标是找到最具传播潜力的独特视角（注意只给出一个切入点即可）：
    A. 观点/洞察视角 (My Perspective):** 如果要针对这个信息发表一篇观点文，最犀利、最能体现我“体系构建师”人设的洞察是什么？
    B. 实操/教程视角 (Practical Methods):** 如果要将这个信息做成一篇保姆级教程，最能帮用户解决实际问题的n8n自动化流程或实操方法是什么？
    C. 解读/对比视角 (In-depth Interpretation):** 如果要写一篇深度解读，可以将它与什么工具、趋势或理念进行对比，从而产生新的认知？]

**4. 内容升维策略 (Content Elevation Strategy)**
    请回答：如何将这个选题从一个“资讯”升维成一个兼具“稀缺性”和“趣味性”的深度内容？可以从哪些角度补充信息，或者结合什么独特的案例来让它变得无可替代？
```

理由：通过这一步的“思考”，为下一步的正式写作定下基调，避免 AI 胡乱发挥。

```

  data = typeof raw === "string" ? JSON.parse(raw) : raw;
} catch (e) {
  // 情况2：如果包含前后文本，尝试用正则提取 JSON 部分
  const match = raw.match(/\{[\s\S]*\}/);
  if (!match) {
    throw new Error("未找到 JSON 字段，请检查上游节点输出");
  }
  data = JSON.parse(match[0]);
}

// 安全读取三个字段
const title = data.Title || data.title || "";
const content = data.Content || data.content || "";
const summary = data.Summary || data.summary || "";

// 返回这三个字段
return [
  {
    json: {
      title,
      content,
      summary
    }
  }
];
```

3.3.7 节点十六：Code （拆分 media\_id） 节点

此节点充当了数据流转过程中的精密过滤器。在上一环节完成封面上传后，微信服务器返回了一段包含时间戳、URL 等冗杂信息的复杂数据包，但后续的文章发布接口仅能识别 media\_id 这一串字符作为素材的唯一身份证。因此，我们需要利用此节点对返回数据进行清洗与提纯，剔除所有无关杂质，精准提取出那个至关重要的 ID 字符串，确保后续环节的调用凭证绝对纯净且有效。

节点配置操作（如下图所示）：

点击 上传封面图片 节点右侧的加号，搜索并选择 Code 节点，这一步旨在利用脚本语言处理非标准化的数据提取任务。

在 JavaScript 代码输入框中，完整复制并填入下方的脚本内容，这一步操作的核心在于建立一套多重容错机制。代码逻辑会依次检测数据是直接存在、嵌套在 data 对象中、还是以字符串形式存在，这种“三层保险”能确保无论 API 返回何种格式，系统都能稳定地解析并输出正确的 media\_id，避免因格式差异导致工作流中断。

•

```
// n8n Code node
return items.map(item => {
  let mediaId = '';

  // 第一种：直接在 json.media_id
  if (typeof item.json?.media_id === 'string') {
    mediaId = item.json.media_id;
  }
  // 第二种：在 json.data.media_id
  else if (typeof item.json?.data === 'object' && typeof item.json.data.media_id === 'string') {
    mediaId = item.json.data.media_id;
  }
  // 第三种：data 是 JSON 字符串
  else if (typeof item.json?.data === 'string') {
    try {
      const parsed = JSON.parse(item.json.data);
      if (parsed && typeof parsed.media_id === 'string') {
        mediaId = parsed.media_id;
      }
    } catch (e) {}
  }

  return {
    json: {
      media_id: mediaId
    }
  };
});
```

此步骤用于物理闭合整个自动化工作流的运行链路。我们将流程末端的执行结果反馈给循环控制器，以此作为当前单次任务彻底结束的信号。这一连接激活了 Loop 节点的迭代机制，使其能够自动释放下一条待处理的数据，从而实现对多篇文章的连续自动化处理。

节点配置操作（如下图所示）：

点击 更新飞书多维表格 节点右侧的输出圆点（Output），按住鼠标左键拖拽出一条连接线，将其直接连接回 Loop Over Items 节点的左侧输入端口（Input），这一步操作旨在构建可视化的循环回路。

配置理由： 这是一个标准的循环结构闭合操作。通过将终点连回起点，明确告知 Loop 节点当前条目的所有处理步骤已全部完成，请求开始下一次迭代。
